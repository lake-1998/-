## 一、设计原则与总体方向

### 1.1 合并后的设计核心矛盾

| 维度 | 小铁寄存柜（线下） | 无忧存（线上） | 合并后策略 |
|------|-----------------|--------------|---------|
| 访问来源 | 扫码80%+（直达下单，不过首页） | 搜索60% | **扫码流直达下单，首页专注搜索找点** |
| 核心诉求 | 扫码→选柜/填信息→支付 | 搜索地标→找网点→预约 | **两条路径独立优化，互不干扰** |
| 用户画像 | 旅游+商务人群 | 30岁以内60% | **年轻化但不失专业感** |
| 信任需求 | 柜子在眼前，不需要 | 线上决策，需要信任背书 | **详情页增强可信度** |

### 1.2 核心路径重新梳理（v2.0 最重要修正）

**路径一：扫码直达（机柜二维码 / 人工网点物料二维码）**
```
用户扫码 → 直接跳转下单页
  ├── 智能寄存柜：选择柜门 → 支付下单
  └── 人工寄存网点：填写存取时间 + 行李数量 → 支付下单
```
> ⚠️ 扫码用户**不会经过首页**，因此首页的布局设计不需要为扫码用户适配。
> v1.0 中的"场景A：扫码进入首页"逻辑错误，已删除。

**路径二：搜索找点（运营入口 / 搜索 / 分享链接 / 直接进入小程序）**
```
进入小程序首页 → 搜索地标/城市/景区 → 地图/列表页 → 网点详情 → 下单页
```
> 首页唯一目的：帮用户快速找到目标寄存网点（人工 + 机柜均支持）。

### 1.3 四大设计原则

1. **路径清晰**：扫码流和搜索流完全独立，首页专注搜索找点，不混淆
2. **关键路径极简**：进入到完成寄存的主路径点击不超过4步
3. **信任前置**：关键转化节点提前展示安全保障（保险、评价、下单后解锁信息）
4. **双端统一**：微信/支付宝/抖音小程序视觉统一，体验一致

### 1.4 设计规范

- **主色调**：小铁黄 `#FFB300`（按钮、强调色、底部导航选中态）
- **辅色**：深灰 `#333333`（主文字）、中灰 `#999999`（辅助文字）、浅灰 `#F5F5F5`（背景）
- **组件库**：沿用小铁寄存柜 UI 组件，融合无忧存业务功能
- **字体**：主文字16px，副文字13px，注释12px
- **圆角**：卡片8px，按钮24px（pill形）

## 二、首页优化方案（核心）🔴

> 首页服务的是**路径二（搜索找点）**的用户，核心目标是让用户最快找到目标寄存网点。

### 2.1 当前问题诊断

```
当前首页结构（存在问题）：
┌─────────────────────────────────┐
│  顶部Banner（可配置）            │
├─────────────────────────────────┤
│  ┌─────┐ ┌─────┐ ┌─────┐        │  ← ❌ 3模块平铺，视觉权重相同

│  └─────┘ └─────┘ └─────┘        │

│  🔍 搜索地标                    │  ← ❌ 4个入口（定位/热门/更多/搜索框）
│  📍 定位  🔥热门推荐  更多>      │    指向同一目的地，冗余

**核心问题**：
- 三大功能模块（扫码/配送/网点）平铺，用户不知该选哪个
- 搜索功能有4个入口（定位、热门推荐标签、更多、搜索框）全部跳同一页面，冗余
- "我的附近"在页面下方，点击率只有4.66%，与搜索差距13倍
- 城市切换放在顶部有兼容性问题（小程序定制受限）

> Banner置顶保障运营引流；搜索和附近网点紧邻，覆盖"有目标"和"看周边"两类用户；金刚区和活动区作为次要内容在下方。

### 2.3 关键设计决策说明

**Banner区**（首屏顶部）：
- 保留Banner，位置上移至顶栏正下方，是重要引流区域
- 轮播图，内容由运营配置（活动/推广/新功能引导）
- ❌ 不做"上滑缩起"动效：手势与页面滚动冲突，实现复杂且易误触

**搜索区**（Banner下方）：
- 搜索框为首页视觉第二焦点，Banner下方紧跟
- 合并原来4个搜索入口为1个（搜索框 + 3个热门标签）
- ✅ 新增**历史搜索缓存**：展示在热门标签下方（最近3条，可清除）
- 历史搜索用 **localStorage 本地存储**，无需登录即可使用；可接受卸载数据丢失

**附近网点区**（搜索区下方，紧邻搜索）：
- 动态显示附近网点数量，右侧"查看地图"快捷跳转
- 横向3张小卡片预览，展示：**真实缩略图（不模糊）+ 距离 + 价格起步**
- 缩略图使用真实网点图片，不做模糊处理（小尺寸本身对爬虫价值极低，模糊反而影响用户第一印象）
- ❌ 评分不作为卡片主要信息：平台绝大部分网点为5.0，对用户无选择参考价值

**金刚区配置规则（v2.0确定）**：
- 单行4个坑位，**第4位固定为"更多"**，点击底部弹出半屏功能列表
- 功能按钮排序（从左至右）：
  1. 📷 扫码存包（固定第1位）
  2. 🚚 行李配送（城市白名单：福州/厦门地铁/杭州地铁；未开通城市由下一位补位）
  3. 🗄️ 长租柜（后续可配置）
  4. 🎫 我的优惠券（补位顺序第1候选）
  5. … 更多功能按 sort_order 排列
  - 第4位固定"更多▼"，不被功能挤占
- 全部功能项均支持后台配置（无需发版），运营可灵活调整排序和显示条件
- 标签规范：4字以内，语义明确（"扫码存包"而非"扫码"）

**底部Tab（3个，v2.0修正）**：
- 首页 / 订单 / 我的
- ❌ 去掉"行程服务"Tab：行程服务放金刚区，底部Tab只放高频核心功能
- 订单Tab有未处理订单时显示红点badge

### 2.4 搜索能力优化

| 能力 | 现状 | 优化 |
|------|------|------|
| 搜索入口 | 4个（定位/热门/更多/搜索框） | 合并为1个搜索框 + 3个热门标签 |
| 历史记录 | 无 | ✅ 新增历史搜索缓存（localStorage本地存储，最近3条，无需登录）|
| 搜索范围 | 支持人工寄存 + 机柜 | 保持不变（混合搜索） |
| 热门标签 | 可变 | 3个固定标签，由运营配置，不展示"更多"按钮 |

## 四、网点详情页优化 🔴🔴🔴

> **现状核实**：实景图、商家电话、店铺名称因防爬/防飞单，**必须下单后才能查看**（非登录后可见）。

### 4.1 防爬虫与防飞单策略（v2.0修正）

**v1.0 方案（已废弃）**：登录后显示精确地址，下单后显示电话
**v2.0 方案（与现实对齐）**：

| 用户状态 | 显示内容 | 隐藏内容 |
|---------|---------|---------|
| 任何状态（未下单） | 区域描述、价格、营业时间、标签、服务数量、保障信息 | 精确地址、商家电话、实景图（大图）、精确店铺名称 |
| 已下单 | 全部信息解锁：精确地址 + 实景图 + 电话 | — |

> ℹ️ 暂不按"未登录/已登录/已下单"三级角色区分权限。分级权限方案已单独记录，留作后续优化阶段参考（见附录）。

**保留文案**（在详情页中明显展示）：
```
下单后我们将提供精确地址和联系方式
```

### 4.1.1 降低用户对"下单才能看"的抵触感（不改变锁定逻辑）

> 当前锁定逻辑不变。以下是在现有约束下，通过文案和设计手段减少用户抵触的优化方向。

**① 地图已显示精确点位，锁定的是导航能力**

> ✅ 现状修正：详情页地图已显示精确网点位置（精确点），未下单时锁定的是**拉起导航的能力**，而非地图位置本身。用户可看到店在哪里，但无法直接在地图内一键导航。

优化建议：在地图上精确点旁边，加一个"🔒 下单后解锁导航"的浅色提示标签，让用户明白"位置我知道了，下单可以一键导过去"——把导航解锁作为下单的正向激励，而非隐藏位置的负面限制。

**② 区域级文字描述代替"模糊地址"**

不是空白，也不是"***"，而是有意义的描述：
- ✅ "外滩附近 · 南京东路步行街区域"
- ✅ "西湖景区南门附近"
- ❌ "地址：****"（显示星号比没有更差）

让用户知道大概在哪，对周边熟悉的用户就能判断是否值得下单。

**③ 将"下单看地址"从限制感改为权益感**

| ❌ 当前表达（限制感）| ✅ 优化表达（权益感）|
|------|------|
| 下单后可见 | 下单后解锁精确导航 |
| 地址：下单后查看 | 📍 下单即获精确导航地址 |
| 商家电话不可见 | 📞 下单后直联商家 |

**④ （已移除）取消政策展示**

> 本期暂不展示"2小时内免费取消"，待业务确认取消政策规则后再评估。

**⑤ 用已服务人数替代地址作为信任背书**

```
🏅 已为 238 位旅客提供寄存服务
```
比精确地址更能建立信任，且完全可以公开显示。

**⑥ 支付成功页第一时间强提示地址解锁**

```
支付成功页（下单后立即出现）：

🎉 预订成功！
📍 精确地址已解锁 → [一键导航]
📞 商家电话：138 **** 8888
```
确保用户感受到"解锁"是立即生效的奖励，而非被忘掉的承诺。

### 4.2 详情页布局优化

```
网点详情页优化布局：

┌─────────────────────────────────────┐

### 4.3 底部说明文案精简原则（v2.1修订）

**现状问题**：详情页底部有大量说明文案（寄存完整流程说明、寄存需知、退改规则、注意事项），导致：
- 用户理解困难，信息过于繁琐
- "违约金"字眼让用户产生不安全感，降低下单意愿

**优化原则**：
1. 寄存须知精简为 **3条核心**（超过寄存时间会产生额外费用 / 贵重物品请自行保管 / 如需延时请提前联系）
2. 退改规则用**正向、时间轴式**表达（详见下方），去掉"违约金"措辞
3. 流程说明移至帮助中心，详情页只保留快速链接入口"查看完整流程"

**退改规则重写（v2.1，已确认触发条件）**：

> 以下为用户端展示文案建议，采用时间轴结构，正向表达为主：

```

│  平台将根据实际使用情况处理退款，可能扣除部分费用 │

┌─ 预约已结束 ──────────────────────────────────┐

```

**文案说明**：
- "预约进行中"期间取消，实际扣除比例约为20%（平台规则），文案不直接写百分比，避免制造心理障碍
- 完全去掉"违约金"一词，改用"可能扣除部分费用"的中性表达
- 规则兜底文案保留：如有疑问，可联系客服处理
